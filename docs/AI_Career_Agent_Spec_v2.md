# AI Career Agent — объединённая спецификация

**Версия:** 2.0
**Статус:** Draft, синтез `SPEC.md` (продуктовое видение) и `README.md` (реализованный движок StarJobs)
**Владелец:** Даниил Александров
**Тип продукта:** личный инструмент (не мультитенантный SaaS)

---

## 0. Что изменилось относительно v1.0

Эта версия — не новая идея, а исправление v1.0 с учётом того, что часть инфраструктуры уже реально существует (StarJobs) и работает по-другому, чем предполагалось. Ключевые правки:

| # | Было в SPEC.md v1.0 | Стало в v2.0 | Почему |
|---|---|---|---|
| 1 | Мультитенантный продукт подразумевался неявно | Явно: личный инструмент, без биллинга и подписок | У тебя один пользователь — ты. Логика Stars-подписки из StarJobs не нужна |
| 2 | Получение вакансий — email-парсинг рассылки HH («Вариант 1») | Прямой `api.hh.ru` | Уже реализовано в StarJobs правильно, парсинг писем — лишний и более хрупкий слой |
| 3 | «Автоматическая отправка отклика» без оговорок | Полу-автоматика: бот готовит, человек кликает финальную кнопку | У HH нет надёжного публичного write-API на отклики; риск ограничения аккаунта |
| 4 | Стек — Python/FastAPI/aiogram/Celery с нуля | Расширение существующего TS-движка StarJobs | Не дублировать уже написанный и рабочий код |
| 5 | География деплоя не упомянута | RU-хостинг или прокси — обязательное требование | `api.hh.ru` блокирует зарубежные дата-центровые IP |
| 6 | Два трека — как будто два независимых бота | Два трека — это два **профиля поиска** в одном движке (число ботов — вопрос UI, не архитектуры) | Экономит дублирование кода, БД, деплоя |

## 1. Цель проекта

Создать персонального AI-агента, который берёт на себя рутину поиска работы:

- ежедневно ищет вакансии по двум трекам;
- анализирует и оценивает каждую через LLM;
- присылает саммари в Telegram с кнопками действий;
- пишет сопроводительное письмо под конкретную вакансию, согласовывает с тобой;
- помогает отвечать работодателям в переписке;
- раз в неделю показывает статистику и подсказывает, какие навыки подтянуть.

Не цель — быть публичным продуктом. Если в будущем захочется сделать SaaS из этого (как сделал приятель со StarJobs) — это отдельное решение, которое потребует отдельного слоя биллинга, изоляции пользователей и, вероятно, отказа от жёсткой привязки к твоему личному резюме. Пока не проектируем под это, чтобы не усложнять MVP.

## 2. Архитектура

Один движок, два трека, общие сервисы. **Не два независимых бота с нуля** — это означало бы дублировать всё: интеграцию с HH, LLM-слой, хранилище, крон. Правильнее:

```
┌─────────────────────────────────────────────┐
│                  Ядро (общее)                 │
│                                               │
│  sources/   — адаптеры источников (hh, ...)  │
│  ai/        — LLM-клиент, скоринг, генерация │
│  jobs/      — pipeline: поиск→анализ→отбор   │
│  store/     — хранилище (вакансии, отклики,  │
│               переписка, резюме-профили)      │
│  resumes/   — версии резюме по трекам        │
└─────────────────────────────────────────────┘
              │                    │
      ┌───────┴──────┐    ┌────────┴───────┐
      │   Track A     │    │   Track B      │
      │ Видео/продакшн│    │  IT PM/PdM     │
      │ фильтры, вес  │    │ фильтры, вес   │
      │ скоринга,     │    │ скоринга,      │
      │ резюме-профиль│    │ резюме-профиль │
      └───────┬───────┘    └────────┬───────┘
              │                     │
      ┌───────┴─────────────────────┴───────┐
      │         Telegram (1 бот или 2)        │
      │   — вопрос UI, не архитектуры ядра    │
      └───────────────────────────────────────┘
```

**Один бот с переключателем трека или два бота (два токена) на общем ядре** — оба варианта технически равноценны, разница только в UX. Два бота проще ментально разделяют «мои текущие вакансии» и «мои IT-вакансии» и не мешают друг другу в ленте сообщений — на старте это, вероятно, удобнее.

## 3. Источники вакансий

### Сейчас (MVP)
- **hh.ru** — `api.hh.ru`, официальный API, без ключа, полный набор фильтров (должность, регион, зарплата, опыт, формат работы, ключевые слова). Основной источник для обоих треков.
- **Трудвсем (opendata.trudvsem.ru)** — открытые госданные, best-effort, полезен как дополнение, не основной.

### Инфраструктурное требование (критично, из README)
`api.hh.ru` блокирует запросы с зарубежных дата-центровых IP (`403 forbidden`) независимо от User-Agent. Значит:
- деплой — на российском хостинге, **или**
- через собственный reverse-proxy на RU-VPS, проксирующий `api.hh.ru`.

Без этого весь трек по HH просто не будет работать при деплое на зарубежном сервере (Railway, Render, Fly.io и т.п. по умолчанию — не RU).

### Roadmap (V3, при необходимости)
- LinkedIn, Habr Career, GeekJob — публичных API нет, потребует парсинга/эвристик, приоритет ниже HH.
- Email-парсинг рассылки HH — **исключено из спеки**, дублирует прямой API и добавляет хрупкость (OAuth к почте, парсинг вёрстки письма, которая может меняться).

## 4. Track A — поиск по текущему опыту

Цель: максимизировать число качественных откликов по специализации «Руководитель проектов» в видео/медиа/продакшн.

Поиск строится вокруг текущего резюме без адаптации терминологии — здесь у тебя прямое совпадение с рынком.

## 5. Track B — переход в IT (Product/Project Manager)

Цель: находить вакансии Product Manager / Project Manager в IT, даже при неполном формальном совпадении, через **transferable skills** — перевод продакшн-опыта на язык IT-менеджмента.

Карта переноса опыта (используется LLM-промптом при скоринге и при генерации сопроводительного письма — важно: это **переформулировка**, не приписывание несуществующих навыков):

| Твой опыт (как есть) | Как звучит на языке IT PM/PdM |
|---|---|
| Вёл 40+ параллельных проектов, pipeline «бриф → выдача → фидбек» | Управление бэклогом/pipeline, координация множественных потоков работ |
| Система заявок через Tally + Notion | Построение внутренних инструментов и процессов, no-code tooling |
| Управлял 30+ людьми в 3 отделах, найм по всему миру | Кросс-функциональное лидерство, международный найм |
| Взаимодействие с маркетингом, legal, аналитикой | Stakeholder management, кросс-департаментная координация |
| Внедрение AI-инструментов в продакшн | AI-tooling adoption — сильный аргумент именно сейчас |
| Запуск на разных платформах (YouTube, Twitch, Kick) | Multi-platform product launches |

Резюме для Track B — отдельный профиль (см. раздел 12), где эти формулировки зафиксированы один раз как базовый шаблон, а не изобретаются заново под каждую вакансию.

## 6. Pipeline

```
Источники (hh.ru, Трудвсем)
        ↓
  Получение вакансий (по фильтрам трека)
        ↓
  Дедупликация (по vacancy_id — вакансия может прийти повторно)
        ↓
  AI-анализ: извлечение полей (компания, зарплата, стек,
  требования, обязанности, опыт, английский)
        ↓
  Скоринг (структурированный LLM-вывод, см. раздел 7)
        ↓
  Фильтр по порогу релевантности
        ↓
  Саммари → Telegram
        ↓
  Пользователь: 👍 откликнуться / ❌ не интересно / ⭐ сохранить / 📄 открыть
        ↓
  (если 👍) Генерация сопроводительного письма → согласование
        ↓
  Подготовленный отклик → отправка человеком (полуавтоматика, раздел 8)
        ↓
  Входящие сообщения от работодателя → уведомление + предложенный ответ
        ↓
  Обновление статуса в истории
        ↓
  Еженедельная аналитика (по понедельникам)
```

## 7. AI-анализ и скоринг вакансии

Из вакансии извлекается структурированными полями (JSON, не свободный текст — это то, что уже правильно сделано в StarJobs через `{relevant, score, reason}`, расширяем до полного набора полей):

```json
{
  "company": "...",
  "salary": "...",
  "region": "...",
  "remote": true,
  "stack": ["..."],
  "requirements": ["..."],
  "responsibilities": ["..."],
  "experience_required": "...",
  "english_level": "...",
  "match_score": 0-100,
  "match_reasons": ["..."],
  "mismatch_reasons": ["..."]
}
```

Скоринг — взвешенная модель (веса настраиваются отдельно на трек, т.к. для Track B «требования» и «навыки» весят больше, а «совпадение опыта» в буквальном смысле — меньше):

| Фактор | Track A (вес) | Track B (вес) |
|---|---|---|
| Совпадение опыта | 35% | 20% |
| Навыки / transferable skills | 20% | 35% |
| Зарплата | 15% | 15% |
| Формат работы | 10% | 10% |
| Индустрия | 10% | 5% |
| Требования (hard-блокеры: язык, виза и т.п.) | 10% | 15% |

LLM-провайдер: для скоринга подходит связка «бесплатная модель → фолбэк на более сильную» как в StarJobs, но для **генерации сопроводительного письма** экономить на модели не стоит — это текст, который увидит работодатель, здесь нужна модель уровня Claude Sonnet/GPT-4-класса, не free-tier.

## 8. Отклики: что реально автоматизируется

Важное уточнение к SPEC v1.0: полностью автоматическая отправка отклика через HH — рискованная затея. У HH нет публичного write-API на отклики, открытого для сторонних приложений (антифрод-ограничения), а браузерная эмуляция авторизованной сессии — серая зона ToS, чреватая ограничением аккаунта.

**Реальный флоу:**
1. Бот готовит сопроводительное письмо под вакансию.
2. Показывает черновик в Telegram с кнопками: `Отправить` / `Сделать короче` / `Сделать официальнее` / `Редактировать`.
3. После твоего подтверждения — два варианта:
   - **Полуавто:** бот открывает ссылку на отклик и присылает готовый текст письма для вставки — ты кликаешь сам через приложение/сайт HH (1 клик, но человеком);
   - **Авто через partner-доступ:** если получится оформить HH как OAuth-приложение с доступом к `negotiations` — тогда можно автоматизировать полностью. Это отдельная задача регистрации приложения у HH, не гарантирована.

MVP закладываем первый вариант, второй — как апгрейд, если доступ получится оформить.

## 9. Саммари вакансии в Telegram

```
[Track A / Track B]  Руководитель проектов

Компания: ООО Ромашка
Зарплата: 250–350 тыс.
Удалёнка: Да
Совпадение: 91%

Почему подходит:
• управление проектами
• Agile
• руководство командой

Что смущает:
• требуется SAP
• английский C1

[краткое описание — 2-3 предложения]

[👍 Откликнуться] [❌ Не интересно] [⭐ Сохранить] [📄 Открыть вакансию]
```

Если вакансия опубликована недавно, у неё мало откликов и высокий match score — помечается 🔥 **Горячая вакансия** и присылается вне обычного расписания, сразу.

## 10. Работа с сообщениями от работодателей

Когда приходит сообщение от работодателя (через переписку по отклику на HH):
1. Уведомление в Telegram с текстом сообщения и контекстом (какая вакансия, какая компания).
2. LLM предлагает 1-2 варианта ответа, опираясь на резюме и историю переписки.
3. Кнопки: `Отправить` / `Изменить` / `Позвонить позже` (то есть отложить и просто напомнить).

Чтение своей переписки на HH (входящие по своим же откликам) — легитимная операция для собственного аккаунта, в отличие от массовой отправки откликов; здесь риск ниже.

## 11. История и статусы

Для каждого работодателя/вакансии хранится: компания, вакансия, история сообщений, история откликов, статус.

Статусы: `Viewed → Responded → Interview → Offer / Rejected / Ignored`

## 12. Версии резюме

Минимум два профиля на старте:

- `resume_track_a.pdf` — как есть, под видео/продакшн.
- `resume_track_b.pdf` — с формулировками из карты переноса опыта (раздел 5), под IT PM/PdM.

Точечная адаптация под конкретную вакансию (`resume_google_pm.pdf`) — не отдельный файл на диске каждый раз, а генерация в моменте перед откликом на основе базового профиля трека + требований вакансии, с твоим подтверждением перед отправкой (та же логика, что и с сопроводительным письмом).

## 13. Еженедельная аналитика (по понедельникам)

```
За неделю:
Просмотрено вакансий: 184 (Track A: 120, Track B: 64)
Откликов: 31
Ответов: 7
Собеседований: 2

Чаще всего требуют (Track B): SQL, Power BI, BPMN
Если подтянуть эти навыки — релевантность вырастет с 72% до ~89%
```

Skill Gap Analysis считается по факту прочитанных за неделю вакансий, а не по общей статистике рынка — это дешевле и точнее для двух конкретных треков.

## 14. V2/V3 — то, что можно отложить

Из v1.0 SPEC, но не критично для MVP:

- Анализ эффективности откликов и автообучение агента (нужна накопленная статистика за месяцы, преждевременно для месяца-двух работы);
- Dashboard с метриками — полезно, но Telegram-дайджеста по понедельникам достаточно для старта;
- Подготовка к интервью (обзор компании, вероятные вопросы, STAR-примеры) — хорошая функция, но отдельный флоу, добавляется после MVP;
- LinkedIn/Habr/GeekJob — низкий приоритет, нет официальных API.

## 15. Стек — рекомендация

Не переписывать с нуля на Python. У тебя (через приятеля) уже есть рабочий движок:

- **Backend:** TypeScript/Node.js — как в StarJobs, расширяем существующие модули `sources/`, `ai/`, `jobs/`.
- **Хранилище:** JSON-файл достаточен для одного пользователя (как сейчас в StarJobs) — Postgres/Redis имеют смысл только если это станет продуктом для многих людей.
- **Telegram:** тот же фреймворк, что уже используется в StarJobs.
- **LLM:** OpenRouter с фолбэком — приемлемо для скоринга; для писем — модель посильнее, не free-tier.
- **Деплой:** RU-хостинг или RU-прокси перед `api.hh.ru` — обязательно.

## 16. Открытые вопросы (нужно решить перед стартом разработки)

1. **Один бот с переключателем трека, или два бота с общим ядром?** Технически равноценно, но UX разный.
2. **Форкать код StarJobs или писать поверх него отдельную ветку/форк?** Зависит от того, готов ли приятель делиться кодовой базой и нужна ли тебе независимость от его продукта.
3. **Оформлять ли OAuth-приложение у HH для доступа к `negotiations`?** Это единственный путь к полной автоматизации отклика — стоит начать процесс регистрации параллельно с разработкой MVP, а не после.
4. **Нужен ли биллинг вообще?** Если это остаётся личным инструментом — нет. Если позже решишь превратить в продукт — это отдельный этап поверх готового ядра, не заранее.
